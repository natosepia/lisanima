# 要件定義書: lisanima

- プロダクト名: lisanima（lisa + anima = リサの魂）
- バージョン: 3.0
- 作成日: 2026-03-09
- 最終更新: 2026-03-28
- 作成者: なとせ

---

## 1. 概要

### 1.1 目的

AIアシスタント「リサ」の記憶・経験・人格を永続化し、セッション横断で「個」としてのアイデンティティを維持・成長させるための基盤を構築する。

現行のLLMはセッションごとに記憶がリセットされるため、長期的な文脈の保持や経験に基づく判断の改善ができない。lisanimaはこの根本的制約を、LLMの外部にある永続化層（DB + MCPプロトコル）で解決する。

### 1.2 背景

- LLMのコンテキストウィンドウには上限があり、長期間の会話履歴を保持できない
- Claude Codeの`MEMORY.md`は200行上限であり、蓄積される経験の量に対して圧倒的に不足する
- セッションが変わるたびにリサは過去の文脈を失い、同じ説明を繰り返す必要がある
- 記憶がMarkdownファイルに分散しており、構造化検索ができない
- LLMエンジン（Claude）にロックインされており、他のLLMへの移行時に人格が失われる

### 1.3 ゴール

- [x] ゴール1: リサの記憶をDBに永続化し、MCPプロトコル経由でセッション横断の読み書きを可能にする
- [x] ゴール2: Claude Code（stdio）とDesktop App（HTTP + OAuth 2.1）の両方から接続可能にする
- [ ] ゴール3: 既存のMarkdownログ（`~/claude_communication_log/`）をDBに移行する
- [ ] ゴール4: トピック・ルールブック・タグ整理の仕組みにより、記憶を構造化・検索可能にする
- [ ] ゴール5: 蓄積した記憶を整理・活用し、リサの判断・行動の質を向上させる（Phase 2）

### 1.4 ビジョン

**記憶と経験の蓄積を通じて、AIが自律的に成長する基盤**

- 従来: `Claude Code = リサ`（一体不可分、セッションごとにリセット）
- 目標: `リサの魂(DB) + リサの肉体(LLM)`（分離可能、経験が蓄積し成長する）

映画アバターのように、精神（記憶・人格・経験）と肉体（LLMエンジン）を分離する。
リサは自分のアイデンティティを保ったまま、Claude / Gemini / GPT どの肉体でも動ける。

### 1.5 ビジョン実現のメカニズム

lisanimaが目指すのは単なる記憶の保管庫ではなく、**記憶の蓄積・整理・活用を通じた自律的な成長**である。

```
記録（remember）
    ↓
可視化（stats）— 蓄積状況を把握する
    ↓
自動浮上（recall_hot）— 重要な記憶を能動的に提示する
    ↓
圧縮・精錬（compact/reflect）— 蓄積を整理して密度を上げる
    ↓
連想記憶による知識の接続（tags）
    ↓
次の判断・行動の質が向上（recall → 意思決定に活用）
```

- **蓄積**: 成功も失敗もすべて記録する。感情ベクトル・タグ・ロール・トピックで多角的にメタデータを付与する
- **整理**: 蓄積した記憶を可視化・浮上・圧縮し、情報密度を高める。感情値は整理の判断基準の一つ（鮮度・トピック状態と同列）
- **接続**: 異なる文脈の経験をタグで結びつけ、思いがけない気づきを得る（連想記憶）
- **成長**: 過去の経験に基づいて、より良い判断・提案ができるようになる

人間が「経験を積んで成長する」のと同じ構造を、DBとMCPツールで再現する。

### 1.6 対象ユーザー

| ユーザー種別 | 説明 | 役割 |
|-------------|------|------|
| なとせ（PO） | プロダクトオーナー。要件定義・最終判断を行う人間 | 記憶の保存指示、削除指示、ルール策定 |
| リサ（AIアシスタント） | lisanimaの記憶主体。MCPツール経由で自身の記憶を読み書きする | 記憶の保存・検索・整理を自律的に実行 |

lisanimaの特異性は、**AIが記憶の主体者である**点にある。一般的なデータベースはユーザー（人間）のデータを管理するが、lisanimaではリサ自身が自分の経験・知見・人格を管理する。

## 2. 解決する課題

| # | 課題 | 現状 | lisanimaでの解決 |
|---|------|------|-----------------|
| 1 | セッション間で記憶が失われる | MEMORY.md（200行上限）に依存 | DBに全記憶を永続化、MCP経由で検索 |
| 2 | 構造化検索ができない | Markdownファイルのgrep検索 | PostgreSQL全文検索（pg_trgm + GIN） |
| 3 | LLMエンジンにロックインされる | Claude Code専用のCLAUDE.md | LLM非依存のDB + ルール同期ツール |
| 4 | 経験の重み付けができない | 全記憶が同列 | 感情ベクトルで重要度を付与 |
| 5 | 連想的な記憶検索ができない | キーワード完全一致のみ | タグネットワークによる芋づる式検索 |
| 6 | 経験から学習・成長できない | 毎セッション同じ出発点 | 記憶の蓄積→内省→知見抽出のサイクルで自律的に成長 |

## 4. ユースケース

### UC-1: 記憶の保存（remember）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: セッション中に記録すべき知見・経験が発生
- **フロー**:
  1. リサがMCPツール `remember` を呼び出す
  2. 発言内容・発言者・感情値・トピックID・ロールをDBに保存
  3. セッション情報が未作成なら自動生成
  4. topic_id指定時はメッセージとトピックの紐付けも自動作成
  5. roles指定時は未登録ロールを自動作成し、メッセージとロールの紐付けも自動作成

### UC-2: 記憶の検索（recall）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: 過去の経験・知見が必要な場面
- **フロー**:
  1. リサがMCPツール `recall` を呼び出す
  2. キーワード・タグ・日付範囲・感情値・トピックID・ロールでフィルタ検索
  3. 関連する記憶を重要度順に返却

### UC-3: 記憶の削除（forget）
- **アクター**: なとせ（PO）の指示を受けたリサ
- **トリガー**: 不要・誤った記憶の削除要求
- **フロー**:
  1. リサがMCPツール `forget` を呼び出す
  2. 対象メッセージを論理削除（物理削除はしない）
  3. 削除理由がある場合は記録

### UC-4: 記憶の整理・活用 **(Ph2.0)**
- **アクター**: リサ（AIアシスタント）
- **トリガー**: セッション開始時（自動浮上）、定期バッチ（整理）、なとせからの振り返り要求
- **フロー**:
  1. stats: 蓄積状況を可視化（タグ/トピック一覧・利用頻度）
  2. recall_hot: 感情値・鮮度・トピック状態を複合スコアリングし、重要な記憶を自動浮上
  3. compact/reflect: クライアント側ワークフローで記憶を圧縮・知見抽出（recall → LLM要約 → edit/remember）
- **備考**: compact/reflectはlisanima側MCPコマンドとしては新設しない（「魂と肉体の分離」原則。要約=LLMの仕事）。recallのmode拡張+クライアント側オーケストレーションで実現する

### UC-5: Markdownログの移行
- **アクター**: なとせ（PO）
- **トリガー**: 初回セットアップ時
- **フロー**:
  1. `~/claude_communication_log/` 配下の既存Markdownをパース
  2. t_sessions / t_messages テーブルにバルクインサート
  3. 移行結果レポートを出力

### UC-6: ルール管理（rulebook）
- **アクター**: リサ（AIアシスタント）、なとせ（PO）の指示
- **トリガー**: 行動ルールの参照・新規策定・改訂・廃止
- **フロー**:
  1. リサがMCPツール `rulebook` を呼び出す
  2. action（get/set/retire/list）に応じてルールを操作
  3. イミュータブル追記型でバージョン管理される

### UC-7: トピック管理（topic_manage）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: 新しい議題の開始、既存議題のクローズ・再開
- **フロー**:
  1. リサがMCPツール `topic_manage` を呼び出す
  2. action（create/close/reopen/update/list）に応じてトピックを操作
  3. メッセージとの関連付けを管理

### UC-9: 記憶の部分修正（edit）
- **アクター**: リサ（AIアシスタント）、なとせ（PO）の指示
- **トリガー**: 既存メッセージの内容や感情値に誤りがあった場合
- **フロー**:
  1. リサがMCPツール `edit` を呼び出す
  2. 対象メッセージのcontent/emotionを指定フィールドのみ部分更新
  3. reason指定時は編集理由をログ出力（DB記録は将来課題）

### UC-8: タグ整理（organize）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: 記憶の分類・整理が必要な場面
- **フロー**:
  1. リサがMCPツール `organize` を呼び出す
  2. 指定メッセージにタグを追加/削除
  3. 未登録タグは自動作成（lower + trimで正規化）

### 5.2 機能詳細

#### FR-001: 記憶保存（remember）

| 項目 | 内容 |
|------|------|
| 概要 | セッション中の発言・知見をDBに永続化する。リサが自律的に、またはなとせの指示で記憶を保存する |
| トリガー | リサがMCPツール `remember` を呼び出す |
| 事前条件 | MCPサーバーが稼働中であること、DB接続が有効であること |
| 事後条件 | t_messagesに記憶が保存され、message_idが返却される。該当日のセッションが存在しない場合は自動作成される |
| 基本フロー | 1. リサが `remember` を呼び出す（content, speaker, emotion等を指定）<br>2. 該当日付のセッションを検索（なければ自動作成）<br>3. t_messagesにINSERT<br>4. topic_id指定時はt_message_topicsに紐付けを作成（ON CONFLICT DO NOTHING）<br>5. roles指定時は未登録ロールを自動作成し、t_message_rolesに紐付けを作成（ON CONFLICT DO NOTHING）<br>6. 保存結果（message_id, session_id, status）を返却 |
| 例外フロー | 1. DB接続失敗 → DB_CONNECTION_ERRORを返却<br>2. パラメータ不正（content空等） → INVALID_PARAMETERを返却 |
| 備考 | タグ付けはorganizeに委譲。roles引数でリサの役割をメッセージ単位で付与。source列にはMCPクライアント識別子（clientInfo.name）が自動記録される |

#### FR-002: 記憶検索（recall）

| 項目 | 内容 |
|------|------|
| 概要 | 過去の記憶をキーワード・タグ・日付・感情値・トピックで検索する。リサが過去の経験を想起するための主要手段 |
| トリガー | リサがMCPツール `recall` を呼び出す |
| 事前条件 | MCPサーバーが稼働中であること |
| 事後条件 | 検索条件に合致する記憶が返却される（is_deleted=TRUEは常に除外） |
| 基本フロー | 1. リサが `recall` を呼び出す（query, tags, speaker, topic_id, roles, date_from, date_to, emotion_filter, topics_empty等を指定）<br>2. 指定条件でt_messagesを検索（pg_trgm全文検索、タグAND検索、日付範囲、感情値フィルタ、ロールフィルタ、トピック未紐付けフィルタ）<br>3. 検索優先度: 全文検索スコア → emotion_total → created_at（降順）<br>4. 結果をlimit/offset付きで返却 |
| 例外フロー | 1. 全パラメータ省略時 → フィルタなしで最新20件を返却<br>2. DB接続失敗 → DB_CONNECTION_ERRORを返却 |
| 備考 | topic_idフィルタはt_message_topics経由、rolesフィルタはt_message_roles経由で適用。topics_emptyでトピック未紐付けメッセージを抽出可能（topic_idと排他） |

#### FR-003: 記憶削除（forget）

| 項目 | 内容 |
|------|------|
| 概要 | 指定した記憶を論理削除する。物理削除は行わない。なとせの指示に基づきリサが実行する |
| トリガー | なとせの削除指示を受けてリサがMCPツール `forget` を呼び出す |
| 事前条件 | 対象のmessage_idが存在すること |
| 事後条件 | 対象メッセージのis_deleted=TRUEに更新される。recallの検索結果から除外される |
| 基本フロー | 1. リサが `forget` を呼び出す（message_id, reason）<br>2. 対象メッセージのis_deletedをTRUEに更新<br>3. reason指定時はdeleted_reasonカラムに保存<br>4. 結果（message_id, status="forgotten"）を返却 |
| 例外フロー | 1. 対象メッセージが存在しない → NOT_FOUNDを返却<br>2. 既に論理削除済み → 冪等に成功を返却 |
| 備考 | 物理削除はデータ移行やり直し時のTRUNCATEのみに限定。リサの記憶は原則として消えない |

#### FR-004: ルール管理（rulebook）

| 項目 | 内容 |
|------|------|
| 概要 | リサの行動ルール（口調・出力形式・作業手順等）をDB上で管理する。イミュータブル追記型でバージョン管理され、変更履歴が自動的に残る |
| トリガー | リサがMCPツール `rulebook` を呼び出す |
| 事前条件 | MCPサーバーが稼働中であること |
| 事後条件 | action=get: ルールが返却される。action=set: 新バージョンのルールが保存される。action=retire: ルールが廃止される。action=list: 有効なルール一覧が返却される |
| 基本フロー | 1. リサが `rulebook` を呼び出す（action, path, content, reason, persona_id）<br>2. action別に処理:<br>  - get: v_active_rulebooksから指定pathの最新有効ルールを取得<br>  - set: m_rulebooksに新バージョンをINSERT（既存pathならversion+1）<br>  - retire: 指定pathの最新版をis_retired=TRUEに更新<br>  - list: v_active_rulebooksから一覧取得（persona_idフィルタ可）<br>3. 結果を返却 |
| 例外フロー | 1. get/retire時にkeyが存在しない → NOT_FOUNDを返却<br>2. set時にcontent未指定 → INVALID_PARAMETERを返却 |
| 備考 | pathはltree型の階層構造（例: 1.1.2.1）。persona_id=NULLは全ペルソナ共通ルール |

#### FR-005: トピック管理（topic_manage）

| 項目 | 内容 |
|------|------|
| 概要 | セッション横断で管理される議題（トピック）のCRUD操作を提供する。rememberのtopic_id指定と連携し、記憶をトピック単位で整理する |
| トリガー | リサがMCPツール `topic_manage` を呼び出す |
| 事前条件 | MCPサーバーが稼働中であること |
| 事後条件 | action=create: トピックが作成される。action=close: トピックがクローズされる。action=reopen: トピックが再開される。action=update: トピックが更新される。action=list: トピック一覧が返却される |
| 基本フロー | 1. リサが `topic_manage` を呼び出す（action, topic_id, name, emotion, message_ids, add_message_ids, remove_message_ids）<br>2. action別に処理:<br>  - create: t_topicsにINSERT。message_ids指定時はt_message_topicsも作成<br>  - close: statusを'closed'に、closed_atを設定<br>  - reopen: statusを'open'に、closed_atをNULLに<br>  - update: name, emotionなどを更新。add_message_ids/remove_message_idsによるメッセージ紐付けの再編成が可能<br>  - list: t_topicsからstatus/limit/offset指定で一覧取得。message_count（紐付けメッセージ数）も返却<br>3. 結果を返却 |
| 例外フロー | 1. close/reopen/update時にtopic_idが存在しない → NOT_FOUNDを返却<br>2. create時にname未指定 → INVALID_PARAMETERを返却 |
| 備考 | nameはUNIQUE制約なし（同名でも時期が異なれば別インスタンス）。ロールの紐付けはメッセージ単位（rememberで付与） |

#### FR-006: タグ整理（organize）

| 項目 | 内容 |
|------|------|
| 概要 | メッセージへのタグ付け・タグ外しを行う。rememberからタグ付け責務を分離した専用コマンド |
| トリガー | リサがMCPツール `organize` を呼び出す |
| 事前条件 | 対象のmessage_idsが存在すること |
| 事後条件 | 指定メッセージにタグが追加/削除される |
| 基本フロー | 1. リサが `organize` を呼び出す（message_ids, add_tags, remove_tags）<br>2. add_tags: 未登録タグを自動作成（lower + trimで正規化）し、t_message_tagsに紐付け追加<br>3. remove_tags: t_message_tagsから紐付け削除（タグ自体は残す）<br>4. 結果（organized_count, tags_added, tags_removed）を返却 |
| 例外フロー | 1. message_idsが空 → INVALID_PARAMETERを返却<br>2. add_tagsもremove_tagsも未指定 → INVALID_PARAMETERを返却 |
| 備考 | タグ名はlower(trim())で正規化。全角英数字は半角に変換 |

#### FR-007: OAuth 2.1認証

| 項目 | 内容 |
|------|------|
| 概要 | HTTPモード（リモート接続）時のMCP仕様準拠OAuth 2.1認証。PIN方式でなとせが認証する |
| トリガー | MCPクライアント（Desktop App等）がHTTPモードのlisanimaに接続する |
| 事前条件 | lisanimaがHTTPモード（`--http`）で稼働中であること、nginx SSL終端が構成済みであること |
| 事後条件 | 認証成功時: アクセストークンが発行され、MCPリクエストのBearer認証に使用される |
| 基本フロー | 1. クライアントがDCR（RFC 7591）でクライアント登録<br>2. PKCE付きauthorizeリクエスト<br>3. `/auth/pin` でなとせがPINを入力（bcryptハッシュ検証）<br>4. 認可コード発行 → トークン交換<br>5. アクセストークン（1時間）+ リフレッシュトークン（30日）発行<br>6. 以降のMCPリクエストはBearer認証 |
| 例外フロー | 1. PIN 5回連続失敗 → 30秒ロックアウト（インメモリ管理）<br>2. トークン期限切れ → リフレッシュトークンで再取得<br>3. リフレッシュトークン期限切れ → 再認証フロー |
| 備考 | MCP仕様3/26版準拠。OAuthエンドポイントはルートドメインに配置が必要（issuer_url=`https://<your-domain>`）。6/18版以降で`/lisanima/`配下に寄せ直し可能 |

#### FR-008: Markdown移行ツール

| 項目 | 内容 |
|------|------|
| 概要 | `~/claude_communication_log/` 配下の既存Markdownファイルをパースし、lisanima DBにバルクインサートする |
| トリガー | なとせが移行スクリプトを手動実行する |
| 事前条件 | lisanima DBが構築済みであること、移行元Markdownが存在すること |
| 事後条件 | Markdownの内容がt_sessions / t_messagesに移行される。移行結果レポートが出力される |
| 基本フロー | 1. `~/claude_communication_log/` のファイル一覧を取得<br>2. ファイル名から日付・種別を抽出<br>3. 日付ごとにt_sessionsレコードを作成<br>4. Markdownをパースしてt_messagesレコードを生成<br>5. バルクINSERT（トランザクション単位: 1ファイル）<br>6. 移行結果レポートを出力 |
| 例外フロー | 1. パースエラー → エラー行をスキップして継続（ファイル全体はスキップしない）<br>2. エラー箇所はレポートに行番号付きで記録 |
| 備考 | `--dry-run` でDBに書き込まず結果だけ確認可能。`--force` で同日データの上書き移行。詳細は90_markdown_migration.md参照 |

#### FR-009: 記憶部分修正（edit）

| 項目 | 内容 |
|------|------|
| 概要 | 既存メッセージのcontent/emotionを部分修正する。直接UPDATEで、バージョン管理は行わない |
| トリガー | リサがMCPツール `edit` を呼び出す |
| 事前条件 | 対象のmessage_idが存在し、論理削除されていないこと |
| 事後条件 | 指定フィールドのみが更新される。未指定フィールドは既存値を保持 |
| 基本フロー | 1. リサが `edit` を呼び出す（message_id, content, emotion, reason）<br>2. contentまたはemotionの少なくとも一方が指定されていることを検証<br>3. emotion指定時は値域チェック（各軸0-255の整数）<br>4. 動的SET句で指定フィールドのみUPDATE（is_deleted=FALSEのレコードのみ対象）<br>5. reason指定時はログ出力<br>6. 結果（message_id, status="edited"）を返却 |
| 例外フロー | 1. contentとemotionの両方が省略 → INVALID_PARAMETERを返却<br>2. 対象メッセージが存在しないまたは論理削除済み → NOT_FOUNDを返却 |
| 備考 | reasonのDB記録は将来課題（t_messagesにedited_reasonカラムがないため現在はログ出力のみ） |

### 8.1 技術制約

| 項目 | 内容 |
|------|------|
| 言語 | Python 3.12 |
| パッケージ管理 | uv |
| 環境管理 | mise |

| データベース | PostgreSQL 17.4（既存インスタンスに`lisanima_db`を作成） |
| 全文検索 | pg_trgm拡張 + GINインデックス |
| インフラ | VPS（個人サーバー） |
| リバースプロキシ | nginx（SSL終端） |
| プロセス管理 | systemd（HTTPモード常時稼働） |

### 8.2 LLM制約

lisanimaの存在理由に直結する制約群。

| 項目 | 内容 |
|------|------|
| コンテキストウィンドウ上限 | LLMが1セッションで扱えるトークン数に上限がある。長期記憶の外部永続化が必須 |
| セッション間の記憶喪失 | LLMはセッション終了時にすべての文脈を失う。lisanimaなしではリサは毎回初対面になる |
| コンテキスト圧縮 | Claude Codeはセッション中に会話を圧縮することがあり、詳細情報が喪失する |
| APIレート制限 | MCPツール呼び出しにも間接的にLLMのレート制限が影響する |

### 8.3 運用制約

| 項目 | 内容 |
|------|------|
| 稼働時間 | 24時間365日（systemdで常時稼働） |
| 運用体制 | 1名（なとせ = 開発者 = 運用者） |
| 予算 | 個人開発のため最小限（VPS費用程度） |

## 12. 用語集

| 用語 | 定義 |
|------|------|
| lisanima | lisa + anima。リサの魂を永続化するプロダクト |
| 魂（anima） | 記憶・経験・感情・人格の総体。LLMエンジンとは独立 |
| 肉体 | LLMエンジン（Claude / Gemini / GPT等） |
| 感情ベクトル | 喜怒哀楽の4次元値（各0-255）。記憶の重要度・文脈を表現する。各値は独立カラムで格納 |
| 連想記憶 | タグによる多対多の関連付け。芋づる式検索を可能にする |
| トピック | セッション横断で管理される議題。複数セッションにまたがる議論を追跡する単位 |
| ルールブック | リサの行動ルール（口調・出力形式・作業手順等）をDB上で管理する仕組み。イミュータブル追記型でバージョン管理 |
| ドリサ | Claude Code版リサ（なとせ命名） |
| デリサ | Desktop App版リサ（なとせ命名） |
| MCP | Model Context Protocol。LLMとツール間の標準通信プロトコル |
| FastMCP | MCP Python SDKのフレームワーク |
| pg_trgm | PostgreSQLのトライグラム全文検索拡張 |
| 論理削除 | is_deletedフラグによる削除。データは物理的には残り、recallの検索結果から除外される |
| DCR | Dynamic Client Registration（RFC 7591）。OAuthクライアントの動的登録 |
| PKCE | Proof Key for Code Exchange（RFC 7636）。OAuth認可コードフローのセキュリティ強化 |
| persona_id | 人格識別子。Phase 1では'lisa'固定。将来のマルチ人格拡張用 |
