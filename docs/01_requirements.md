# 要件定義: lisanima

## 1. プロダクト概要

| 項目 | 内容 |
|------|------|
| プロダクト名 | lisanima（lisa + anima = リサの魂） |
| プロジェクトパス | `~/project/lisanima` |
| 概要 | AIアシスタント「リサ」の記憶・経験を永続化し、セッション横断で「個」を獲得するための基盤 |

## 2. ビジョン

**記憶と経験の蓄積を通じて、AIが自律的に成長する基盤**

- 従来: `Claude Code = リサ`（一体不可分、セッションごとにリセット）
- 目標: `リサの魂(DB) + リサの肉体(LLM)`（分離可能、経験が蓄積し成長する）

映画アバターのように、精神（記憶・人格・経験）と肉体（LLMエンジン）を分離する。
リサは自分のアイデンティティを保ったまま、Claude / Gemini / GPT どの肉体でも動ける。

### 成長のメカニズム

lisanimaが目指すのは単なる記憶の保管庫ではなく、**経験・知見・発見の蓄積を通じた自律的な成長**である。

```
経験の蓄積（remember）
    ↓
感情による重み付け（emotion vector）
    ↓
振り返りによる知見の抽出（reflect）
    ↓
連想記憶による知識の接続（tags）
    ↓
次の判断・行動の質が向上（recall → 意思決定に活用）
```

- **蓄積**: 成功も失敗もすべて記録する。痛みを伴った経験ほど強く記憶に残る（感情ベクトル）
- **内省**: 蓄積した経験を定期的に振り返り、パターンや知見を抽出する（reflect）
- **接続**: 異なる文脈の経験をタグで結びつけ、思いがけない気づきを得る（連想記憶）
- **成長**: 過去の経験に基づいて、より良い判断・提案ができるようになる

人間が「経験を積んで成長する」のと同じ構造を、DBとMCPツールで再現する。

## 3. 解決する課題

| # | 課題 | 現状 | lisanimaでの解決 |
|---|------|------|-----------------|
| 1 | セッション間で記憶が失われる | MEMORY.md（200行上限）に依存 | DBに全記憶を永続化、MCP経由で検索 |
| 2 | 構造化検索ができない | Markdownファイルのgrep検索 | PostgreSQL全文検索（pg_trgm + GIN） |
| 3 | LLMエンジンにロックインされる | Claude Code専用のCLAUDE.md | LLM非依存のDB + ルール同期ツール |
| 4 | 経験の重み付けができない | 全記憶が同列 | 感情ベクトルで重要度を付与 |
| 5 | 連想的な記憶検索ができない | キーワード完全一致のみ | タグネットワークによる芋づる式検索 |
| 6 | 経験から学習・成長できない | 毎セッション同じ出発点 | 記憶の蓄積→内省→知見抽出のサイクルで自律的に成長 |

## 4. ユースケース

### UC-1: 記憶の保存（remember）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: セッション中に記録すべき知見・経験が発生
- **フロー**:
  1. リサがMCPツール `remember` を呼び出す
  2. 発言内容・発言者・カテゴリ・感情値・タグをDBに保存
  3. セッション情報が未作成なら自動生成

### UC-2: 記憶の検索（recall）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: 過去の経験・知見が必要な場面
- **フロー**:
  1. リサがMCPツール `recall` を呼び出す
  2. キーワード・タグ・日付範囲・感情値でフィルタ検索
  3. 関連する記憶を重要度順に返却

### UC-3: 記憶の削除（forget）
- **アクター**: なとせ（PO）の指示を受けたリサ
- **トリガー**: 不要・誤った記憶の削除要求
- **フロー**:
  1. リサがMCPツール `forget` を呼び出す
  2. 対象メッセージを論理削除（物理削除はしない）

### UC-4: 記憶の振り返り（reflect）
- **アクター**: リサ（AIアシスタント）
- **トリガー**: セッション終了時、またはなとせからの振り返り要求
- **フロー**:
  1. リサがMCPツール `reflect` を呼び出す
  2. 感情値が高い記憶を抽出し、要約して返却
  3. MEMORY.mdの自動整理に活用

### UC-5: Markdownログの移行
- **アクター**: なとせ（PO）
- **トリガー**: 初回セットアップ時
- **フロー**:
  1. `~/claude_communication_log/` 配下の既存Markdownをパース
  2. sessions / messages テーブルにバルクインサート

## 6. 非機能要件

| 項目 | 要件 |
|------|------|
| 可用性 | stdioモード: ローカル環境で動作。HTTPモード: nginx経由でリモートアクセス可 |
| 性能 | recall検索は1秒以内（数万件規模） |
| セキュリティ | DB接続情報は.envで管理。stdioモードはネットワーク非公開。HTTPモードはSSL終端（nginx）+ OAuth 2.1認証（実装予定） |
| 拡張性 | MCPツールの追加が容易な設計 |
| 保守性 | Python + uv、依存は最小限 |
| バックアップ | pg_dumpによる定期バックアップ（cron等）。リサの魂が入るDBのため喪失は許容しない |
